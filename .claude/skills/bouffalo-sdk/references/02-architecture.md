# SDK 架构设计

## 架构概览

Bouffalo SDK 采用分层模块化架构，从应用层到硬件层逐层抽象，实现了代码复用和跨平台兼容性。

```
┌─────────────────────────────────────────────────────────┐
│                    应用层                                │
│              (examples/, 用户代码)                       │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                  组件层 (Components)                     │
│  文件系统、网络协议栈、多媒体、图形库、加密、无线协议等    │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│               板级支持包 (BSP)                          │
│          板级初始化、引脚配置、时钟配置                   │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│         低级别硬件抽象层 (LHAL - 统一API)                │
│         跨芯片统一的硬件驱动接口                         │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│           SOC驱动层 (drivers/soc/)                      │
│         芯片特定的硬件驱动实现                           │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                    硬件层                                │
│              Bouffalo芯片硬件                            │
└─────────────────────────────────────────────────────────┘
```

## 核心设计理念

### 1. LHAL（低级别硬件抽象层）

LHAL是SDK的核心抽象层，为所有Bouffalo芯片提供统一的外设驱动API。

**设计目标：**
- 统一API：所有芯片使用相同的接口
- 易于移植：应用代码可在不同芯片间快速移植
- 最小开销：轻量级抽象，性能损失最小
- 向后兼容：新芯片保持API兼容

**架构层次：**
```
应用层
   ↓
LHAL统一API (bflb_xxx_init/control/io)
   ↓
设备表 (Device Table)
   ↓
芯片配置层 (drivers/lhal/config/blxxx/)
   ↓
SOC驱动层 (drivers/soc/blxxx/std/)
   ↓
硬件寄存器
```

### 2. 组件化系统

**组件分类：**

| 类别 | 组件 | 功能 |
|-----|------|------|
| 操作系统 | FreeRTOS, NuttX | 实时内核、任务调度 |
| 网络 | LwIP, MQTT | TCP/IP协议栈、消息队列 |
| 文件系统 | FatFS, LittleFS, ROMFS | 存储管理 |
| 加密 | mbedTLS | AES、SHA、TLS/SSL |
| 无线 | WiFi, BLE, Zigbee, Thread | 无线通信 |
| 图形 | LVGL v8/v9 | GUI框架 |
| 多媒体 | 音视频编解码 | 音频、视频处理 |
| USB | CherryUSB | USB协议栈 |
| IPC | OpenAMP, RPMsg | 核间通信 |
| 工具 | cJSON, partition, shell | 通用工具 |

**组件特性：**
- 模块化：每个组件独立配置
- 依赖管理：Kconfig自动处理依赖
- 可选配置：通过menuconfig启用/禁用

### 3. 板级抽象（BSP）

**BSP职责：**
- 硬件初始化（时钟、引脚、内存）
- 板级配置（LED、按键、串口）
- 内置镜像（Boot2、PTAB）

**BSP目录结构：**
```
bsp/
├── board/              # 板级配置
│   ├── bl602/          # BL602开发板
│   ├── bl702/          # BL702开发板
│   ├── bl616/          # BL616开发板
│   └── bl618dg/        # BL618DG开发板
└── common/             # 通用BSP驱动
    ├── console/        # 控制台
    ├── flash/          # Flash驱动
    └── utils/          # 工具函数
```

## 构建系统架构

### 混合构建系统

SDK使用Makefile作为入口，CMake作为实际构建引擎，Kconfig作为配置系统。

**构建流程：**
```
Makefile (入口)
   ↓
project.build (Makefile包装器)
   ↓
CMakeLists.txt (CMake配置)
   ↓
board.build (板级配置)
   ↓
Kconfig (配置系统)
   ↓
Ninja/Make (实际构建)
```

### 配置系统

**Kconfig特点：**
- 图形化配置：menuconfig/guiconfig
- 依赖管理：自动处理组件依赖
- 灵活配置：支持defconfig

**配置流程：**
```
menuconfig → .config
   ↓
kconfig2cmake → .config.cmake
   ↓
CMake → CMakeCache.txt
   ↓
Build
```

```
WiFi_MGMR ──┐
             ├──→ COEX ──→ MACSW
NET_AL   ───┤
BT_API   ───┘
```

## 多核架构

对于多核芯片（BL618DG），SDK支持：

### 核心分工

| 核心 | 用途 | 频率 |
|-----|------|------|
| M0 | 主控、通用计算 | 高频 |
| D0 | WiFi/无线专用 | 高频 |
| LP | 低功耗待机 | 低频 |

### 核间通信

**OpenAMP + RPMsg:**
```
┌──────────────┐  RPMsg  ┌──────────────┐
│   M0 Core    │ ←────→  │   D0 Core    │
│  (主控)      │         │  (无线)       │
└──────────────┘         └──────────────┘
      ↑                         ↑
   libmetal                 libmetal
      ↑                         ↑
  共享内存 ←────────→ 共享内存
```

**IPC使用场景：**
- WiFi连接状态通知
- 音视频数据传输
- 多核任务协调

## 内存管理架构

### 内存层次

```
应用内存堆 (TLSF/Freertos Heap)
   ↓
PSRAM (外部，可选)
   ↓
SRAM (内部，高速)
   ↓
Flash (代码存储，XIP)
```

### 内存管理方案

**方案1: TLSF**
- 特点：实时性、低碎片
- 适用：FreeRTOS
- 位置：components/mm/tlsf/

**方案2: FreeRTOS Heap**
- 特点：简单、可靠
- 适用：通用场景
- 位置：os/freertos/

**方案3: NuttX MM**
- 特点：虚拟内存支持
- 适用：复杂应用
- 位置：os/nuttx/

## 安全架构

### 硬件安全引擎

```
应用层
   ↓
Crypto API (mbedTLS)
   ↓
SEC Driver (LHAL)
   ↓
硬件加速引擎
   ├── AES     (加密)
   ├── SHA     (哈希)
   ├── TRNG    (真随机数)
   └── PKA     (公钥算法)
```

### 安全特性

- **安全启动**: 验证固件签名
- **安全OTA**: HTTPS + 签名验证
- **Flash加密**: 数据加密存储
- **密钥保护**: 硬件密钥存储（EFUSE）
- **抗回滚**: 版本回滚保护

## 电源管理架构

### 低功耗模式

```
运行模式
   ↓
浅睡眠
   ↓
深度睡眠
   ↓
休眠
   ↓
关机
```

### 睡眠策略

**自动睡眠:**
- 基于定时器
- 基于外设事件
- WiFi动态调频

**手动睡眠:**
- API调用控制
- 条件唤醒

## 设计模式总结

### 1. 分层抽象
- 每层职责清晰
- 接口稳定
- 实现可替换

### 2. 组件化
- 独立编译
- 按需配置
- 依赖管理

### 3. 可移植性
- 统一API
- 平台抽象
- 最小依赖

### 4. 可扩展性
- 插件式组件
- 驱动框架
- 板级扩展

这种架构设计使得Bouffalo SDK既保持了高性能，又提供了良好的开发体验和跨平台能力。
